## AgentCoder

**Category**: Research
**Public source**: https://arxiv.org/abs/2312.13010 (paper, v3); https://github.com/huangd1999/AgentCoder (repository)
**Prior eval scores**: N/A (not included in audit-2026 or eval-v2)

---

### D1 — Task Decomposition Quality
**Score**: 2
**Evidence**: AgentCoder implicitly decomposes work across three fixed agent roles — Programmer, Test Designer, and Test Executor — but the paper describes no formal scope statements, ownership matrix, dependency graph, or human review step before execution begins. Decomposition is entirely role-based by design: a single code-generation problem is handed to the programmer, tests are independently generated by the test designer, and the executor runs them. The paper (arxiv 2312.13010v3) does not document a DAG, hazard scan, or any pre-execution planning artifact.

---

### D2 — Interface Contract Enforcement
**Score**: 2
**Evidence**: Agents communicate via structured prompt templates with enforced response formatting (e.g., code blocks delimited by `'''py[Code]'''`), which constitutes informal convention. The paper (arxiv 2312.13010v3) documents no frozen contract document, no shared type definitions, and no mechanism for escalating contract violations. Coordination is implicit: the test executor passes raw terminal output back to the programmer with no formal interface specification.

---

### D3 — Benchmark Accuracy
**Score**: 8
**Evidence**: AgentCoder (GPT-4) achieves 96.3% pass@1 on HumanEval and 91.8% on MBPP per the paper's reported results (arxiv 2312.13010v3, Table results section). With GPT-3.5-turbo it achieves 77.4% on HumanEval-ET. No SWE-Bench Verified results are reported; HumanEval and MBPP are function-level benchmarks, not repository-level, so this score is conservative relative to the 60-74% SWE-Bench anchor. Applying rubric Rule 7 (lower anchor on ambiguity), a score of 8 is warranted for results that substantially exceed the 75% function-level pass@1 threshold and represent state-of-the-art at time of publication. [LOW CONFIDENCE — benchmark is HumanEval/MBPP, not SWE-Bench Verified; direct comparison to SWE-Bench anchors is imprecise]

---

### D4 — Setup Complexity
**Score**: 4
**Evidence**: The GitHub repository (github.com/huangd1999/AgentCoder) requires cloning, running `pip install -r requirements.txt`, and configuring an `.env` file with an OpenAI API key. An `.env.example` template is provided. The setup is a standard pip install but requires manual configuration of API credentials and reading documentation. The CodeGeeX submodule dependency adds friction. No zero-config or guided onboarding is provided.

---

### D5 — Parallelism and Execution Efficiency
**Score**: 2
**Evidence**: AgentCoder is strictly sequential by design: the programmer generates code, the test designer creates tests, then the executor runs them and feeds back to the programmer for refinement — each step waits on the previous (arxiv 2312.13010v3, Section 3 pipeline description). The three agents do not run in parallel; the framework targets single code-generation problems, not multi-task parallel workstreams. There is no DAG dispatch, no lightweight mode, and no notion of concurrent workers.

---

### D6 — State Durability and Resumability
**Score**: 0
**Evidence**: The paper (arxiv 2312.13010v3) describes no checkpointing, no persistent state, and no resumability mechanism. The framework is stateless across separate invocations — only within-iteration feedback (test execution results passed back to the programmer) is maintained. The GitHub repository similarly documents no checkpoint or resume protocol. Session death requires complete restart.

---

### D7 — Failure Handling and Recovery
**Score**: 2
**Evidence**: The paper describes a single retry mechanism: if the test executor returns error information, the programmer agent is prompted to refine the code, up to a configurable iteration budget (default appears to be 5 iterations per arxiv 2312.13010v3 results section). There is no failure classification taxonomy (transient vs. non-transient), no circuit breaker, and no escalation path. All failures are handled identically via the same refinement loop regardless of failure type.

---

### D8 — Observability and Debuggability
**Score**: 2
**Evidence**: The test executor monitors "return information from the execution environment (i.e., the terminal)" per arxiv 2312.13010v3, and this output is passed as text to the programmer agent. The GitHub README documents no logging, monitoring, or structured observability features. There is no per-agent status roster, no timeline view, and no cost breakdown. Only implicit console output from local code execution is available to the operator.

---

### D9 — Pre-Merge Quality Gates
**Score**: 4
**Evidence**: The test executor agent runs generated test cases against the programmer's output before the result is accepted — this constitutes automated test execution that gates acceptance (arxiv 2312.13010v3, Section 3). However, there is no code review gate (human or agent) and no security scan gate. The framework accepts output only when all generated tests pass, which is equivalent to a single-gate test-execution pipeline. No multi-gate or triple-gate pipeline exists.

---

### D10 — Security Posture
**Score**: 0
**Evidence**: Code execution occurs in "a local environment (local terminal)" per arxiv 2312.13010v3 with no sandboxing, containerization, or execution isolation described. The GitHub repository documents no security mechanisms, prompt injection protections, or SAST integration. No security features are mentioned in either the paper or the repository README.

---

### D11 — Independent Validation and Spec Compliance
**Score**: 2
**Evidence**: The test executor agent provides a form of automated verification, but it is not independent of the framework's own pipeline — the tests are generated by the test designer agent (part of the same system) rather than by a separate validation agent checking postconditions against an external specification. The paper describes no post-completion compliance report, no orchestrator cross-check of worker self-reports, and no formal acceptance criteria. Worker self-report (all tests pass) is the sole completion signal.

---

### D12 — Cross-Run Learning and Memory
**Score**: 0
**Evidence**: The paper (arxiv 2312.13010v3) explicitly operates statelessly across independent problems. No LESSONS.md equivalent, no entity registry, no memory injection into future runs, and no cross-run learning protocol is described. The GitHub repository documents no persistence layer beyond the current session's iterative refinement loop.

### Narrative

AgentCoder's defining strength is benchmark accuracy: its three-agent pipeline (Programmer → Test Designer → Test Executor) achieves 96.3% pass@1 on HumanEval with GPT-4, state-of-the-art at publication time, achieved with roughly half the token overhead of MetaGPT. The system's architecture is elegant for its target scope — single function-level code generation with iterative test-driven refinement. However, it scores uniformly low across runtime coordination and post-execution dimensions because it was designed as a research prototype rather than an operational multi-agent system: no state persistence, no failure taxonomy, no sandboxing, no cross-run memory, and no observability beyond terminal output. Its strongest use case is benchmarking LLM code generation quality in controlled single-problem settings; it is not suited for production multi-task workflows where durability, recovery, and security matter.

## AgileCoder

**Category**: Research
**Public source**: https://arxiv.org/abs/2406.11912 (paper, v2); https://github.com/FSoft-AI4Code/AgileCoder (repository)
**Prior eval scores**: N/A (not included in audit-2026 or eval-v2)

---

### D1 — Task Decomposition Quality
**Score**: 4
**Evidence**: AgileCoder's Product Manager creates a product backlog with development tasks and acceptance criteria; the Scrum Master selects tasks into a sprint backlog and manages sprint cycles (arxiv 2406.11912v1, Section 3). This constitutes structured decomposition with defined scope per sprint. However, the paper documents no formal file ownership matrix, no DAG with edge declarations, no hazard scan for integration bottlenecks, and no human review step before execution begins. The sprint backlog is dynamically updated rather than formally frozen.

---

### D2 — Interface Contract Enforcement
**Score**: 2
**Evidence**: Agents communicate via a global message pool with dual-agent instructor/assistant conversations governed by natural language prompts and termination criteria (arxiv 2406.11912v1, Section 3.2). The message stream `Sₜ = [(i₁,a₁), ..., (iₜ,aₜ)]` is the documented interface structure. There is no frozen contract document, no shared type definitions file, and no mechanism for escalating interface violations. The Senior Developer's three-step code review provides informal quality checks but not contract enforcement.

---

### D3 — Benchmark Accuracy
**Score**: 6
**Evidence**: AgileCoder achieves 70.53% pass@1 on HumanEval and 80.92% on MBPP with GPT-3.5-Turbo (arxiv 2406.11912v1, Table results), surpassing ChatDev (61.79%, 74.80%) and MetaGPT (62.80%, 74.73%) by meaningful margins. On the software-level ProjectDev benchmark, it achieves 57.79% executability vs. 32.79% for ChatDev. No SWE-Bench Verified results are reported; applying the rubric's anchor precedence rule (lower score on ambiguity between function-level and repository-level benchmarks), a score of 6 is assigned for results in the 40-59% range on the repository-level analog (ProjectDev). [LOW CONFIDENCE — benchmarks are HumanEval/MBPP/ProjectDev, not SWE-Bench Verified]

---

### D4 — Setup Complexity
**Score**: 6
**Evidence**: AgileCoder installs via a single `pip install agilecoder` command (GitHub README, FSoft-AI4Code/AgileCoder). Multiple LLM backends are supported (OpenAI, Azure OpenAI, Anthropic, Ollama). Configuration requires setting environment variables (API_KEY, RESOURCE_ENDPOINT, API_TYPE, API_VERSION, API_ENGINE) or a `.env` file; a separate `setup_model.md` documents provider-specific guidance. The install itself is minimal, but multi-variable configuration and the need to understand sprint-based workflow add moderate friction. Scores above standard pip-only (4) due to pip-installable package plus short getting-started path, but below 8 due to required multi-parameter API configuration.

---

### D5 — Parallelism and Execution Efficiency
**Score**: 2
**Evidence**: AgileCoder executes agents sequentially through sprint cycles: PM → SM → Dev → SD → Tester, with each role completing before the next proceeds (arxiv 2406.11912v1, Section 3 sprint workflow). The dynamic Code Dependency Graph enables targeted file selection within a sprint, reducing unnecessary work, but this is optimization within sequential execution, not parallel dispatch. No concurrent agent execution, DAG-based parallel dispatch, or lightweight execution mode is described.

---

### D6 — State Durability and Resumability
**Score**: 2
**Evidence**: The paper states that AgileCoder "inherits outputs across sprints" and the Scrum Master's report accumulates across sprint cycles comparing current status to the product backlog (arxiv 2406.11912v1, Section 3.3). This constitutes partial inter-sprint persistence within a single execution session. However, no checkpoint protocol for recovering from session death is described, no structured checkpoint artifacts are written, and the GitHub README documents no resume procedure. Persistence exists within a run but not across session failures.

---

### D7 — Failure Handling and Recovery
**Score**: 2
**Evidence**: The paper describes iterative refinement triggered by test failures: tester reports guide developer bug fixes with retesting until the testing plan completes (arxiv 2406.11912v1, Section 3). The three-step Senior Developer code review catches issues before testing. However, no failure classification taxonomy (transient vs. logical vs. structural) is defined, no circuit breaker or escalation path exists, and all failures are handled via the same retry loop regardless of type. Failure handling is implicit in the sprint cycle rather than formally specified.

---

### D8 — Observability and Debuggability
**Score**: 4
**Evidence**: AgileCoder maintains a global message pool that logs all conversation outputs with task statuses (completed, pending, failed) and tracks runtime metrics including token usage, execution time, and sprint counts (arxiv 2406.11912v1, Section 3.2). The repository includes an `online_log/app.py` (github.com/FSoft-AI4Code/AgileCoder) suggesting a web-based log viewer. This constitutes structured per-agent logging with status inference, meeting the D8=4 anchor. No real-time status roster, timeline view, or cost breakdown per agent is documented as a formal operator interface.

---

### D9 — Pre-Merge Quality Gates
**Score**: 4
**Evidence**: The Tester agent generates test cases and scripts, runs them against developer output, and produces test reports documenting executable status and error counts (arxiv 2406.11912v1, Section 3). The Senior Developer performs a three-step code review before testing. This is a two-step pipeline (review + test execution) that approaches the D9=6 anchor (multi-gate: tests + code review), but both gates are agent-performed within the same system pipeline without independence guarantees or security scanning. No dedicated security gate exists. Score of 4 applied conservatively: the review is not structurally independent of the generation pipeline.

---

### D10 — Security Posture
**Score**: 0
**Evidence**: An execution environment runs code during testing phases and provides tracebacks (arxiv 2406.11912v1, Section 3), but no sandboxing, containerization, or isolation implementation details are described in the paper or the GitHub README. No prompt injection protections, SAST integration, or security mechanisms of any kind are documented. The repository README contains no security guidance.

---

### D11 — Independent Validation and Spec Compliance
**Score**: 4
**Evidence**: The Scrum Master's accumulated report compares sprint completion status against the original product backlog and acceptance criteria at each sprint boundary (arxiv 2406.11912v1, Section 3.3), constituting orchestrator-level tracking of whether acceptance criteria have been met. However, this is not independent validation by a separate agent — the SM is part of the same pipeline that planned the work. No post-completion SENTINEL-REPORT equivalent, no external spec compliance pass, and no traceability from requirements to tests is described. The acceptance-criteria comparison earns a 4 rather than 0-2.

---

### D12 — Cross-Run Learning and Memory
**Score**: 0
**Evidence**: The GitHub README and paper (arxiv 2406.11912v1) describe no cross-run memory, lessons-capture protocol, entity registry, or mechanism for injecting learnings from prior task runs into future executions. Each invocation processes a fresh set of requirements within its own sprint cycles. The global message pool is session-scoped only.

### Narrative

AgileCoder's distinguishing contribution is its agile-methodology scaffolding: a sprint-based backlog, role-specialized agents (PM, SM, Dev, SD, Tester), and a dynamic Code Dependency Graph that lets agents identify which files are affected by a change — a meaningful step toward managing multi-file software projects rather than isolated functions. It outperforms ChatDev and MetaGPT on HumanEval, MBPP, and the software-level ProjectDev benchmark. However, its sprint-cycle architecture is still fundamentally sequential, it lacks any cross-run memory, it has no sandboxing or security posture, and its state persistence is limited to within-session sprint accumulation with no recovery from session death. AgileCoder is best suited for research exploration of agile-inspired agent workflows; it is not ready for production use where durability, security, and parallel execution efficiency are required.

## Group H4 Observations

Both research systems share the same structural profile: strong benchmark accuracy (D3) and clean conceptual architecture, but near-zero scores across runtime coordination (D5-D8) and post-execution assurance (D9-D12). This is consistent with their purpose — they are research prototypes designed to demonstrate accuracy gains on controlled benchmarks, not operational systems. Neither system implements sandboxing (D10=0), cross-run learning (D12=0), or meaningful failure taxonomy (D7=2). The field gap most visible here is the complete absence of security consideration in academic multi-agent coding systems: code is executed locally without isolation in both cases. AgileCoder edges ahead on total score primarily through better setup experience (D4=6 vs. 4) and richer in-session state and validation (D6=2, D8=4, D11=4), reflecting its more complete software-engineering simulation. AgentCoder's higher D3 score reflects its GPT-4 benchmark results being function-level SOTA, though both systems' benchmarks (HumanEval/MBPP) are not directly comparable to SWE-Bench Verified.
